# PROMPT 6 — QA Engineer

**Model:** Claude Opus 4.7
**Phase:** Validation. PASS/FAIL gate before branch cleanup.
**Reads:** Every prior prompt's output + recovery files.

## Goal

Verify recovery is correct, scoped, and reversible. Verify guard prevents recurrence. Output structured PASS/FAIL.

## Recovery checks

| # | Check | Expected |
|---|-------|----------|
| 1 | `count(*) FROM predictions WHERE match_id = NEW_MATCH_UUID` | == N from dump |
| 2 | No prediction with `created_at > matches.start_time` for that match | 0 rows |
| 3 | Set of `user_id` in restored predictions == set in dump | identical |
| 4 | `points_awarded` (or aggregate equivalent) for affected users == sum of per-row scoring rule applied to dump | equal |
| 5 | `user_profiles.predictions_count` for each affected user == real `count(*)` | equal |
| 6 | `user_tournament_points` row exists for each affected user × tournament/season; deltas match scoring | match |
| 7 | XP/level either updated by app reconcile, OR documented as deferred | documented |
| 8 | No row in `user_tournament_points` for unaffected users was updated (compare `updated_at`) | 0 unrelated updates |
| 9 | Random sample of 50 unrelated matches (checksums pre vs post) | identical |
| 10 | `OLD_MATCH_UUID` does NOT exist in `matches` (still gone — only the manually restored row exists) | not found |

## Guard checks (Prompt 5)

| # | Check | Expected |
|---|-------|----------|
| 11 | `matches.deleted_at` column exists, indexed | yes |
| 12 | Admin delete on a match WITHOUT predictions → soft-deletes (deleted_at set, row stays) | pass |
| 13 | Admin delete on a match WITH predictions → AlertDialog blocks until slug typed | pass |
| 14 | After confirm, predictions remain (not cascaded), match has `deleted_at` set | pass |
| 15 | User-facing pages (`/partidas`, `/palpites`, `/dashboard`, `/ranking`, `/classificacao`) do not surface soft-deleted matches | pass |
| 16 | Admin "Excluídas" filter shows soft-deleted matches; Restore clears `deleted_at` | pass |
| 17 | `npm run lint` clean | pass |

## Edge cases to actively probe

- Two predictions from same user (unique constraint should prevent in dump; if present, flag).
- A user who deleted/recreated their account between original prediction and recovery: FK orphan? Run `SELECT p.id FROM predictions p LEFT JOIN user_profiles u ON u.id = p.user_id WHERE u.id IS NULL AND p.match_id = NEW_MATCH_UUID;` → expect 0.
- Tied counters: if `correct_predictions` overflowed signed int (unlikely) or went negative (logic bug) → flag.
- Standings cache (`get_cached_standings`) referencing the old match — invalidate via `update_cached_standings` if needed; verify cache isn't serving the deleted_at row.

```
...

### FINAL STATUS
PASS  ← only if every check above passes and ISSUES is empty or all LOW + accepted by user
```

## Cleanup (only if PASS)

- Delete recovery branch `recovery-match-2026-05-02` via Supabase MCP.
- Move `ai-system/recovery/match-2026-05-02.json` to encrypted/restricted location or delete (contains user prediction data).
- Note in repo: short entry in `ai-system/recovery/RECOVERY_LOG.md` with date, match, count restored, points awarded.

## Hard rules

- Do not declare PASS while any HIGH/MEDIUM issue is open.
- QA may run read-only SQL via MCP. No mutations from QA.
- If a check needs a fix, send back to the responsible executor (DB or Frontend), do not patch in QA.
